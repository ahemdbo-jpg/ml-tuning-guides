estimator
Type: scikit-learn model object
REQUIRED

What It Does:

The machine learning model you want to optimize. This is the base model that will be trained with different hyperparameter combinations.

Syntax:
estimator=RandomForestClassifier()
üìö Common Examples:
RandomForestClassifier()
For classification tasks with ensemble methods
LogisticRegression()
For binary or multiclass classification
SVC()
Support Vector Machine for classification
XGBClassifier()
Gradient boosting for complex patterns
üí°
Pro Tips:
  • Always instantiate the model (include parentheses): RandomForestClassifier() not RandomForestClassifier
  • Don't set hyperparameters here - they'll be overwritten by grid search
  • You can set non-tuned parameters: RandomForestClassifier(random_state=42)

param_grid / param_distributions
Type: dict or list of dicts
REQUIRED

What It Does:

param_grid (GridSearchCV): Dictionary defining exact values to test for each hyperparameter.

param_distributions (RandomizedSearchCV): Dictionary defining distributions or ranges to sample from.

GridSearchCV Syntax:

  'n_estimators': [50, 100, 200],
  'max_depth': [5, 10, 15]
}
RandomizedSearchCV Syntax:
from scipy.stats import randint
param_distributions = {
  'n_estimators': randint(50, 500),
  'max_depth': randint(5, 50)
}

cv
Type: int or cross-validation generator
IMPORTANT

What It Does:

Number of folds for cross-validation. Splits your training data into K parts, trains on K-1 parts and validates on the remaining part. Repeats K times so every part serves as validation once. This gives a more reliable estimate of model performance.

Syntax:
cv=5   # Most common choice

üéõÔ∏è Common Values:
cv=3
Faster, less reliable. Use for quick experiments or large datasets
cv=5 ‚≠ê
DEFAULT - Best balance between speed and reliability
cv=10
More reliable but slower. Use for small datasets
cv=StratifiedKFold(5)
Maintains class distribution in each fold (better for imbalanced data)
üí°
Pro Tips:
  • Higher cv = more reliable but slower. Each combination is trained cv times!
  • For small datasets (< 1000 samples), use cv=10
  • For large datasets (> 100K samples), use cv=3 to save time
  • For imbalanced classes, use StratifiedKFold to maintain class ratios
‚ö†Ô∏è
Common Pitfalls:
  • cv=2 is too low - results will be unreliable
  • cv=20 is usually overkill and wastes compute time
  • Don't forget: Total training time = combinations √ó cv √ó training_time_per_model

scoring
Type: string or callable
IMPORTANT

What It Does:

The metric used to evaluate model performance and determine which hyperparameters are "best". This is CRITICAL - choosing the wrong metric can lead to a model that optimizes for the wrong thing!

Syntax:
scoring='accuracy'   # For classification
scoring='f1'   # For imbalanced classes
scoring='roc_auc'   # For probability-based models

‚úó Bad for imbalanced data

‚úì Balances precision & recall

üìà Classification Metrics:
'accuracy'
Overall correctness. Use when classes are balanced.
'f1' / 'f1_weighted'
Harmonic mean of precision & recall. Best for imbalanced data.
'precision' / 'recall'
Use when false positives or false negatives are more critical.
'roc_auc'
Area under ROC curve. Great for binary classification.
üìâ Regression Metrics:
'neg_mean_squared_error'
MSE (negative because sklearn maximizes scores). Penalizes large errors heavily.
'neg_mean_absolute_error'
MAE (negative). More robust to outliers than MSE.
'r2'
R¬≤ score. Measures proportion of variance explained (0 to 1).
üí°
Choosing the Right Metric:
  • Fraud Detection: Use 'recall' or 'f1' - missing fraud is worse than false alarms
  • Spam Detection: Use 'precision' or 'f1' - false positives annoy users
  • Medical Diagnosis: Use 'recall' - missing a disease is critical
  • Balanced Classes: 'accuracy' is fine
  • Multiple Metrics: Can pass a list: scoring=['accuracy', 'f1', 'roc_auc']

  • On some systems, n_jobs=-1 with certain models can cause memory issues if you have limited RAM
  • Windows users: parallel processing can be slower due to process spawning overhead

verbose
Type: int (0, 1, 2, or 3)
OPTIONAL

What It Does:

Controls how much progress information is printed during the search. Higher numbers = more detailed output. Useful for monitoring long-running searches!

Syntax:
verbose=0   # Silent (default)
verbose=1   # Minimal output
verbose=2   # Detailed output (recommended)
verbose=3   # Very detailed (can be overwhelming)

üéõÔ∏è Verbosity Levels:

Silent. No output. Use for production or when you don't need updates.

Shows just the initial setup info. Minimal.

  • Output goes to stdout - redirect to a log file: python script.py > output.log 2>&1

n_iter
Type: int (RandomizedSearchCV only)
IMPORTANT

What It Does:

RandomizedSearchCV ONLY: Number of random parameter combinations to sample. This controls the tradeoff between search time and thoroughness.

Syntax:
n_iter=100   # Sample 100 random combinations
üéØ n_iter Impact on Search Coverage

n_iter=10

Sparse sampling
‚è±Ô∏è Fast but might miss good regions

n_iter=100 ‚≠ê

Good coverage
‚ö° Best balance

n_iter=1000

Dense sampling
üêå Slow, diminishing returns

üéØ Recommended Values:
n_iter=10-20
Quick exploration. Use for rapid prototyping or initial experiments.
n_iter=50-100 ‚≠ê
RECOMMENDED for most cases. Good balance of thoroughness and speed.
n_iter=200-500
Thorough search. Use for final optimization or production models.
n_iter=1000+
Very thorough but slow. Usually overkill - diminishing returns.
üí°
Pro Tips:
  • Start with n_iter=20 for quick experiments, then increase if needed
  • Rule of thumb: n_iter should be ~10-20% of total possible combinations in param_distributions
  • More iterations = more time but better chance of finding good hyperparameters
  • With cv=5 and n_iter=100, you'll train 500 models total (100 √ó 5)

random_state
Type: int or None (RandomizedSearchCV only)
OPTIONAL (but recommended!)

What It Does:

RandomizedSearchCV ONLY: Seed for the random number generator. Makes your random search reproducible - you'll get the same random samples every time you run it!

Syntax:
random_state=42   # Use any integer (42 is popular by convention)
random_state=None   # Different results each run (default)

Run 2:
Best: n_est=283, depth=12 ‚Üí 88.9%

Run 3:
Best: n_est=283, depth=12 ‚Üí 88.9%

‚úÖ Same results every time

Reproducible experiments
Easier to debug

üí°
Pro Tips:
  • ALWAYS set random_state=42 (or any number) for reproducibility
  • 42 is popular by convention (from "Hitchhiker's Guide to the Galaxy"), but any integer works
  • Essential for: debugging, comparing methods, writing papers, production pipelines
  • Only affects RandomizedSearchCV - GridSearchCV is deterministic by nature
‚ö†Ô∏è
Common Pitfalls:
  • Forgetting random_state makes results non-reproducible - disaster for scientific work!
  • Different random_state values give different results - don't cherry-pick!
  • Still need to set random_state in your model too: RandomForestClassifier(random_state=42)

üîß Recommended GridSearchCV Setup

For most common use cases

from sklearn.model_selection import GridSearchCV

grid = GridSearchCV(
    estimator=YourModel(),
    param_grid=your_param_grid,
    cv=5,              # 5-fold CV
    scoring='accuracy',   # Or f1, roc_auc, etc
    n_jobs=-1,          # Use all cores!
    verbose=2           # Show progress
)

grid.fit(X_train, y_train)

üé≤ Recommended RandomizedSearchCV Setup

For most common use cases

from sklearn.model_selection import RandomizedSearchCV

random = RandomizedSearchCV(
    estimator=YourModel(),
    param_distributions=your_param_dist,
    n_iter=100,         # Test 100 combinations
    cv=5,              # 5-fold CV
    scoring='accuracy',   # Or f1, roc_auc, etc
    n_jobs=-1,          # Use all cores!
    random_state=42,    # Reproducibility!
    verbose=2           # Show progress
)

random.fit(X_train, y_train)
